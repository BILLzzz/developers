Basics

The feed is hosted at Amazon in Ireland and there is no limt on how often you download it. But generally we recommend you to cache the result in an sutiable way for you so that the user does not have to wait for the server to download the feed. In all the samples we use caching in some kind and as far as possible we describe other caching solution as well. How long you do want to cache is completely op to you, but keep in mind that the reviews, trustscore evt. does only update as often as you do. So caching to long will result in the data not being updated often enough, and to often result in bader performance for the user. An sample feed can be download from here http://s.trustpilot.com/tpelements/917278/f.json.gz. This feed is for the demo webshop www.demoshop.com. You can see the reviews on this domain here Demoshop and you are also welcome to create more reviews on this site as it is for demo usage only.

Search engine optimization

If you want to show your Trustpilot reviews on our webpage the widget is a great and easy way of showing them. But the widget is an iFrame solution and therefor you will not be any SEO juice from all the nice reviews. For that reason Trustpilot also provide a JSON feed with the same information as the widget just in JSON format. This feed can be used to build the same widget and just for the propose of getting SEO for the reviews or it can be used for building a widget that support you design the way you thing it will best support your business.

There has been build some samples in different languages. If your favorite language is not ammong them you are welcome to push i to the source and we will add it with the other samples with all credits to you.

C# .NET Sample

Here is an sample of how to download the JSON feed and parsing it to an set of models in C# .NET.

The sample uses System.Web.Caching for caching the result as this is highly recommende to use some kind of cache. Caching in this situation makes sure that the user does not have to wait for the server to download the feed and parse it every time.

There is several other way of doing this caching, one way could be just to download the feed and save it to local disk, and then every ones in a will download the file again. You could also parse this to you database and then get the data from there at every request. In all cases it could be an good idea to have an service running in the background that does fetch and parse (if need) the feed. This has the advantage that the user never has to wait for the data to download and parse. This is how ever not necessary as the feed normally is not bigger then 3-4KB.

If the FeedModel is not in the cache the code downloads the feed directly from the server. It then deserializes the string downloaded to an the FeedModel created for the purpose. The FeedModel can then be used to show the data in the frontend.
